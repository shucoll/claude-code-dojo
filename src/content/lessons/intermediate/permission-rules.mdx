# Permission rules

## The problem

In <LessonLink id="I1.1" /> you learned that every permission prompt is a tool asking, and the tier predicts which prompts you get. But you are still clicking those prompts one at a time, session after session: yes to `npm test`, yes to the same edit, yes to `git status` you have approved a hundred times. Clicking works until it doesn't. It trains you to approve without reading, which is the habit that lets a bad action slip through, and it interrupts you for things you were never actually worried about.

Permission rules turn that clicking into a written policy. You allow the safe things once, deny the dangerous things always, and keep the prompt for the few actions that genuinely deserve a pause.

## The concept

A permission rule is one line in a list. There are three lists, named for what they do:

- **`allow`**: run without asking.
- **`deny`**: never run, block it outright.
- **`ask`**: always prompt, even when something else would have auto-approved.

Every tool call is checked against all three, in a fixed order: **deny first, then ask, then allow.** The first list that matches decides the outcome, and it stops there. So deny beats ask beats allow, always. Rule *specificity* does not change this: a broad `deny` beats a precise `allow`, because the bucket it lives in is checked first.

That single sentence, **deny > ask > allow**, explains almost every permission surprise you will ever have.

Rules live in your settings files and you manage them with the `/permissions` command, which lists every active rule and the file it came from. There are three files, and <LessonLink id="I1.3">the next lesson</LessonLink> maps them properly; for now the short version is enough:

| File | Who it is for |
|---|---|
| `.claude/settings.json` | the team, committed to the repo |
| `.claude/settings.local.json` | just you, gitignored |
| `~/.claude/settings.json` | you, across every project |

## How it works

A rule is either a bare tool name or `Tool(specifier)`. The bare name matches every use of the tool; the specifier narrows it. And here the risk tiers from <LessonLink id="I1.1" /> pay off directly, because each tier has its own specifier shape.

<ChartEmbed id="permission-rule-shapes" />

**Runs commands (Bash).** Command patterns with `*` as a wildcard that can sit anywhere:

- `Bash(npm run build)` matches that exact command.
- `Bash(npm run *)` matches any command starting with `npm run `.
- `Bash(* --version)` matches any command ending in ` --version`.

One detail carries most of the confusion: **the space before `*` enforces a word boundary.** `Bash(ls *)` matches `ls -la` but not `lsof`; `Bash(ls*)` matches both. Claude Code is also shell-aware, so a rule has to match *each* subcommand of a compound command independently: `Bash(safe-cmd *)` does not authorize `safe-cmd && rm -rf .`. Read-only commands (`ls`, `cat`, `git status`, and the rest of the built-in set) never need a rule; they never prompt in the first place.

**Reads and writes files (Read, Edit).** Paths follow [gitignore](https://git-scm.com/docs/gitignore) syntax, not regex. `Edit` rules cover every file-editing tool; `Read` rules cover Grep, Glob, and `@`-mentions too. The anchor is the part people miss:

- `Read(.env)` matches any `.env` at or below the current directory (a bare name matches at any depth).
- `Edit(src/**)` matches everything under `src/`.
- `/path` anchors to the *settings file's* location, not the filesystem root. In project settings, `Edit(/src/**)` means `<project>/src`.
- `//path` is a real absolute path; `~/path` is home-relative.

**Reaches the network (WebFetch).** A `domain:` prefix matched against the hostname:

- `WebFetch(domain:example.com)` matches that host.
- `WebFetch(domain:*.example.com)` matches any subdomain, but not `example.com` itself.

**MCP tools** (a forward look at <LessonLink id="I5.1">Module I5</LessonLink>'s territory) use `mcp__server__tool`, or `mcp__server` for every tool a server exposes.

### How rules meet modes

Permission *modes* from <LessonLink id="B3.3" /> set the broad posture for a session; rules are the fine-grained layer underneath. The two combine along one clean line: **deny and ask rules always hold; modes and allow rules can only ever loosen, never override them.** `acceptEdits` auto-accepts edits, but a `deny` on a path still blocks it. `bypassPermissions` skips prompts, but an explicit `ask` rule still stops to ask (and `rm -rf /` still trips the circuit breaker). Think of modes as how hard you are leaning on the gas, and deny/ask rules as walls that do not move.

## Walkthrough: a starter policy

Here is a policy worth typing into a real project. It allows your everyday commands, keeps one risky-but-routine action on a prompt, and hard-blocks the things you never want touched.

```json
{
  "permissions": {
    "allow": [
      "Bash(npm test)",
      "Bash(npm run *)",
      "Edit(src/**)"
    ],
    "ask": [
      "Bash(git push *)"
    ],
    "deny": [
      "Read(.env)",
      "Read(.env.*)",
      "Bash(rm -rf *)"
    ]
  }
}
```

Read it top to bottom, but remember Claude Code reads it deny-first:

- The **deny** list wins every tie. `Read(.env)` and `Read(.env.*)` stop Claude reading your secrets even though nothing here allows it, which makes the intent explicit and survives someone later adding a broad `Read` allow. `Bash(rm -rf *)` blocks the obvious recursive delete.
- The **ask** list is for actions that are fine most of the time but that you want to see. `git push *` leaves your machine and rewrites a shared branch, so it prompts every time, and because ask beats allow, it keeps prompting even if you later allow `Bash(git *)` wholesale.
- The **allow** list clears the daily friction: your test command, your run scripts, and edits inside `src/`.

You do not have to hand-write this. The fastest way to grow a policy is to click **"Yes, don't ask again"** on a prompt you trust; Claude Code writes the matching rule for you (a command prefix for Bash, the exact path for a file). Run `/permissions` any time to see what has accumulated and from which file, and to add or remove rules directly.

## Use-case scenarios

- **You approve `npm test` every single session.** Add `Bash(npm test)` to `allow`. It runs silently from then on, in that project.
- **You never want Claude reading credentials.** Add `Read(.env)` and `Read(.env.*)` to `deny`. Reliable against honest mistakes, though not a vault (see below).
- **Edits in `src/` are fine, but migrations are sacred.** `allow` `Edit(src/**)` and `deny` `Edit(**/migrations/**)`. The deny wins wherever the two overlap, so you get freedom in `src/` and a wall around migrations.
- **You want Claude to fetch your framework's docs but nothing else.** Add `WebFetch(domain:docs.example.com)` to `allow`. Other domains keep prompting.
- **Near-miss:** you add `Bash(git *)` to `allow` to stop git prompts, and now `git push` runs unattended too. The fix is not to narrow the allow, it is to add `Bash(git push *)` to `ask`. Ask beats allow, so the broad convenience stays and the one dangerous verb comes back under your eye.

## When to use / when not to use

**Reach for a rule when** the decision is stable: a command you approve constantly, an action you always forbid, a convention your whole team should share (put those in the committed `.claude/settings.json`). Rules also cost essentially nothing: they are enforced by the Claude Code harness, not loaded into the model's context, so a large policy is free in a way a large `CLAUDE.md` is not (a distinction <LessonLink id="I1.1">the tool belt</LessonLink> hinted at and I2 makes central).

**Do not reach for a rule when:**

- **You need a real guarantee against evasion.** Argument-constraining Bash denies are fragile: `rm` has many spellings, `curl` follows redirects, a variable can hide a URL. A deny rule is a strong speed bump against mistakes, not a wall against a determined or prompt-injected adversary. For that, use a `PreToolUse` hook (<LessonLink id="I4.1">Module I4</LessonLink>) or the OS-level sandbox (Advanced).
- **It is a one-off.** A command you will run once does not need a policy. Just click.
- **You would be allowing broadly to save typing.** A bare `Bash` allow or a wide `Bash(git *)` trades away the safety you came for. Scope allows narrowly and let `ask` cover the exceptions.

## Pitfalls

These are the rule-syntax surprises that cost people the most time.

- **The space before `*`.** `Bash(ls *)` and `Bash(ls*)` are different rules: the first requires a word boundary and matches only `ls`, the second also matches `lsof`, `lsblk`, and anything else starting `ls`.
- **It is not regex.** Paths are gitignore patterns and Bash is glob matching. A `.` is literal, `*` spans one path segment (or, in Bash, any run of characters), `**` crosses directories. Writing `Read(.*env)` expecting regex will not do what you think.
- **`/path` is not absolute.** A single leading slash anchors to the settings file's own location. `Edit(/src)` in project settings means `<project>/src`, not `/src` on your disk. Use `//` for a true absolute path.
- **A deny cannot carry an exception.** `deny: Bash(aws *)` blocks every `aws` call, including one that also matches `allow: Bash(aws s3 ls)`. Deny is all-or-nothing; if you need a hole, do not deny that broadly.
- **Bare name versus scoped deny behave differently.** `deny: Bash` removes Bash from Claude entirely, so it never even tries to use it. `deny: Bash(rm *)` leaves Bash available and blocks only matching calls. Denying a whole tool by name is a bigger hammer than it looks.
- **Compound commands split.** A rule is checked against each subcommand. Approving `git status && npm test` saves a rule for `npm test`, and a `Bash(safe *)` allow never covers `safe && something-else`.

## Why did it ask? Why didn't it?

Most real-world confusion is not "how do I write a rule" but "why did Claude Code do the opposite of what I expected." A short playbook:

1. **Run `/permissions` first.** It lists every active rule *and the file each came from*. The usual culprit is a rule you forgot in another scope, especially a user-level or managed one.
2. **Recite the order: deny > ask > allow.** If it asked despite your allow, hunt for a matching `ask` rule (or a mode). If it blocked, a `deny` matched somewhere. If it ran silently and you did not expect it, an `allow` matched, or the command is on the built-in read-only list.
3. **Remember scope precedence.** A deny at *any* level beats an allow at *any* level. A user-level or organization-managed deny will override your project's allow, and no local setting can loosen it.
4. **Read the pattern literally.** Re-check the space-before-`*` boundary, the `/path` anchor, and glob-not-regex. Launch with `--verbose` to see the exact command string and parameters Claude actually sent, which is often not quite what you pictured.
5. **"Why didn't it ask before `git status`?"** Because read-only commands never prompt in any mode. If you genuinely want a prompt for one, add it to `ask`.

On a Bash prompt, `Ctrl+E` also explains what the command does and rates its risk, which settles a lot of "should I approve this" questions before you touch a rule.

## Try It

**Environment:** any git repository, either a small public repo you cloned or a throwaway scratch directory with `git init`. Everything here goes in `.claude/settings.local.json` (your personal, gitignored file) and acts only on a scratch file you create, so it will not touch the repo's own config or history.

1. Open `/permissions`, or create `.claude/settings.local.json` with an empty `permissions` block.
2. **Allow:** add `Bash(touch *)`. Ask Claude to `touch scratch-test.txt`. It should create the file with no prompt.
3. **Deny:** add `Bash(rm *)`. Ask Claude to delete `scratch-test.txt`. It should refuse and tell you a rule blocked it, rather than prompting.
4. **Ask:** add `Bash(git status)`. Run `git status` through Claude. A command that is normally silent now stops to ask you, which shows an `ask` rule overriding the read-only default.
5. Run `/permissions` and confirm all three rules appear, each tagged with `settings.local.json` as its source.

**Success criteria:** the allow ran silently, the deny refused without a prompt, the ask forced a prompt on a normally-automatic command, and you can point to where each rule lives.

## FAQ

**Where exactly do rules live?** In the three settings files above, and you edit them with `/permissions` or by hand. Which file to choose (team vs personal vs global) is the subject of <LessonLink id="I1.3">the next lesson</LessonLink>, on settings and scopes.

**Can a rule use a regular expression?** No. Bash rules are glob patterns and file rules are gitignore patterns. This trips up anyone expecting regex; there is no `.*` or anchors, just `*` and `**`.

**I allowed a command and it still asked. Why?** Something with higher precedence matched: an `ask` rule, a `deny` rule, or a mode. Run `/permissions` and apply deny > ask > allow. Also check that your allow pattern actually matches (the space-before-`*` boundary is the usual miss).

**Do permission rules cost me context?** No. They are enforced by the harness and never enter the model's context window, unlike `CLAUDE.md` instructions. You can keep a large policy at no token cost.

**Is a deny rule real security?** It reliably stops honest mistakes and keeps Claude away from files and commands you have marked off. It is not proof against evasion or prompt injection. When you need a hard guarantee, a `PreToolUse` hook (<LessonLink id="I4.1">Module I4</LessonLink>) or the sandbox is the right tool.

**How do I remove a rule?** Delete it in `/permissions`, or remove the line from the settings file. Rules take effect immediately.
