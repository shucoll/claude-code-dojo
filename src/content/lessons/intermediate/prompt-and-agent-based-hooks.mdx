# Prompt-based and agent-based hooks

## When a shell command is not enough

Every hook so far has been a shell command: it reads the event JSON, runs some logic, and decides. That works whenever the decision is something a script can compute. "Is this path under `migrations/`?" is a string match. "Is this command an `rm -rf`?" is a grep.

Some conditions are not computable that way. "Does this edit weaken test coverage?" "Is this commit message actually descriptive?" "Is this a safe place to write a file?" A regex cannot answer those, because the answer needs judgment. For those, a hook can hand the decision to a model instead of a script. That is the same `type` field you already set to `command`, pointing at one of two more powerful, more expensive handlers.

The rule this lesson lands on: **shell when you can, prompt when you need judgment, agent only when the judgment needs tools.** The rungs go up in capability and up in cost at the same rate, so the discipline is to stay on the lowest rung that answers the question.

## Rung one: the shell hook

The one you know. `type: "command"`, decides by running code, costs only execution time and no tokens. If the condition is expressible as a script, this is the answer, and the other two rungs are a mistake. A shell hook runs locally, returns instantly, and spends nothing on the model. Reach past it only when a script genuinely cannot make the call.

## Rung two: the prompt hook

When the decision needs a model to read and weigh, `type: "prompt"` sends your prompt plus the event JSON to a Claude model for a single-turn yes/no. No script, no tools, just a judgment.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "prompt",
            "prompt": "Should Claude run this command: $ARGUMENTS\n\nConsider security implications.",
            "model": "claude-opus-4-1-20250805",
            "timeout": 30
          }
        ]
      }
    ]
  }
}
```

`$ARGUMENTS` is where the event JSON lands. The model must reply with a fixed shape:

```json
{ "ok": true, "reason": "Optional explanation" }
```

`ok: true` proceeds, `ok: false` blocks, and what "blocks" means is the same per-event story from <LessonLink id="I4.2" />: on `PreToolUse` the tool is denied, on `Stop` Claude cannot stop, on `PostToolUse` the reason is just shown. The `model` field is optional and defaults to a fast model; name a stronger one only when the judgment needs it. Everything else about the hook, the event, the matcher, the `if` clause, is unchanged from a shell hook. Only the handler is different.

## Rung three: the agent hook

When the judgment needs to look at more than the event JSON, when it has to read files, search the codebase, or reason across several steps, `type: "agent"` spawns a subagent that can use tools (Read, Grep, Glob) before it decides.

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "agent",
            "prompt": "A file is about to be written to $ARGUMENTS.\n\nCheck whether the target directory exists and is a safe location. Deny if it is a protected system directory.",
            "timeout": 60
          }
        ]
      }
    ]
  }
}
```

It returns the same `{ "ok", "reason" }` shape and follows the same per-event blocking rules. The difference is that it can gather evidence first, which is what lets it answer questions a single prompt cannot. Agent hooks are **experimental**, so treat the config and capabilities as subject to change.

## The cost that decides the rung

A shell hook spends nothing on the model. The moment you climb to a prompt or agent hook, every firing of that event spends tokens and latency, because a model runs each time. This is the rent from <LessonLink id="I2.1" />, now billed per event rather than per session.

That matters most on hot events. A prompt hook on `PreToolUse` runs a model inference before potentially every tool call in a turn. An agent hook is worse: its latency scales with the number of tool calls it makes to verify the condition, so a single guard can add several seconds and a real bill to every edit. On a once-per-turn `Stop` hook the cost is bounded; on a per-tool-call `PreToolUse` hook it compounds.

So the ladder is a cost ladder as much as a capability one:

| Rung | Decides by | Costs | Use when |
|---|---|---|---|
| Shell (`command`) | running a script | execution time only | the condition is computable |
| Prompt | one model judgment | tokens + latency, every fire | judgment needed, no tools |
| Agent | a model with tools | tokens + latency, scaling with tool calls | judgment needs to inspect files |

Climb only when the rung below cannot answer. A prompt hook doing string matching a shell hook could have done is pure waste; an agent hook doing what a prompt could is worse.

## Try It

**Environment:** a project where a few Bash commands are routine.

1. Add the prompt hook from rung two to `~/.claude/settings.json` (drop the `model` field to use the default fast model).
2. Ask Claude to run a harmless command like `ls`. It should proceed, since the model returns `ok: true`.
3. Ask it to run something the prompt would flag on security grounds. Watch it get blocked with the model's `reason`.
4. Notice the latency: each command now waits on a model inference before running. That pause is the cost the table is about.

**Success looks like:** a hook that blocks on a judgment no regex could express, and a felt sense of what it costs on every fire, the trade-off that decides whether to use one.
