# Where you are, where you're going

## You are here

You finished Beginner by shipping Shelf and writing a `WORKFLOW.md` in your own words: the loop you now run without thinking about it. You can scaffold a project with intent, design a multi-file change in plan mode before a line is written, break something and rewind without fear, ask for proof instead of hoping a change works, keep a long session healthy when context fills, and ship a repo you would sign your name to. That is genuine fluency with Claude Code as it comes out of the box, and everything in this level takes it as the starting line.

## The gap

Out of the box has a ceiling, and you have probably already met it:

- You explain the same convention every session, and a busy one forgets it anyway. Re-typing "use pnpm, run the tests before committing" is not a workflow, it is a tax you pay every morning.
- You need a guarantee, not a polite request. "Never touch the migrations" written in CLAUDE.md is advice Claude can miss under context pressure, and some rules have to hold every single time.
- Claude cannot see the systems your work actually lives in: your GitHub issues, your database, your error tracker. So you copy from a browser tab it cannot reach and paste the results back by hand.

Each of these has a purpose-built answer, and this level is the guided tour of them. The theme in one sentence: Beginner was about *using* Claude Code well, and Intermediate is about *shaping* it.

## How it fits together

Two of these layers are foundations and the rest are choices. <LessonLink id="I1.1">Tools, permissions, and settings (I1)</LessonLink> are the substrate: every other feature is assembled from the same tools you already watched Claude use, and all of it answers to the same permission machinery. <LessonLink id="I2.1">Context engineering (I2)</LessonLink> is the economy: every feature you add spends some of Claude's context before you type a word, so the budget quietly frames every later decision.

On top of that foundation sit four mechanisms, each answering a different question. <LessonLink id="I3.1">Skills (I3)</LessonLink> teach Claude a repeatable procedure. <LessonLink id="I4.1">Hooks (I4)</LessonLink> enforce something deterministically, with no model judgment involved. <LessonLink id="I5.1">MCP (I5)</LessonLink> connects Claude to an outside service. <LessonLink id="I6.1">Subagents (I6)</LessonLink> run heavy work in an isolated context and hand back only a summary. They are not rivals competing for the same job; recognizing which one a given problem calls for is the recurring skill of this level, and every module ends by resolving that question against its neighbors. <LessonLink id="I7.1">Plugins (I7)</LessonLink> are the packaging layer that bundles whatever you assembled into one installable unit.

## Suggested route

Take <LessonLink id="I1.1">I1</LessonLink> and <LessonLink id="I2.1">I2</LessonLink> **first**. They are not optional background: every other module leans on the tool belt, the permission rules, and the context budget they establish. After that, <LessonLink id="I3.1">I3</LessonLink> through <LessonLink id="I6.1">I6</LessonLink> come in **any order**. Start with whichever gap above bit you hardest, since each mechanism stands on its own. Save <LessonLink id="I7.1">I7</LessonLink> for **last**, because it packages what the earlier modules produce and there is nothing to bundle until they exist. This overview and the <LessonLink id="I7.5">end-of-level self-assessment</LessonLink> are both skippable, though the assessment is the cheapest way to find out which "which mechanism?" decision you have not yet made second nature.

## What you'll build

The level lands in a guided project, <LessonLink id="I8.1">PulseBoard</LessonLink>: a GitHub project-health dashboard built with skills, hooks, an MCP connection, and a review subagent all working together. The twist that makes it an Intermediate project rather than a Beginner one is that the configured environment is itself a deliverable. You finish with two things: the working dashboard, and a `pulseboard-dev` plugin any teammate could install to stand up the same setup in one command. The loop you wrote into `WORKFLOW.md` at the end of <LessonLink id="B5.6" /> still runs underneath all of it. This level just gives it sharper tools.
