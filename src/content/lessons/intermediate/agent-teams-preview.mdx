# Agent teams: preview

Subagents gave you parallel work in <LessonLink id="I6.3" />: fan out several, each in its own window, each reporting back to you. There is a structurally different way to run many Claudes at once, and the intermediate map has a gap until you can place it.

This is a preview, not a how-to. It gives you enough to answer "should I care about agent teams yet?" and no more.

## The concept

An agent team is several Claude Code sessions working as peers, sharing a task list and messaging each other directly.

One session is the lead. It spawns the others, but it does not sit between them: teammates claim tasks off a shared list, talk to each other, and challenge each other's findings without routing through the lead. Compare that to a subagent, which is an isolated delegate that can only report back to whoever spawned it and never sees another subagent.

The difference is a topology. Subagents are spokes around a hub. A team is a mesh.

## Against subagents

| | Subagents | Agent teams |
|---|---|---|
| **Context** | own window; result returns to the caller | own window; fully independent |
| **Communication** | reports back to the caller only | teammates message each other directly |
| **Coordination** | the caller manages all work | shared task list, teammates self-claim |
| **Best for** | focused work where only the result matters | work that needs discussion between the workers |
| **Token cost** | lower; the result is summarized back | higher; each teammate is a full Claude instance |

The deciding row is communication. If your parallel workers never need to hear from each other, subagents already do the job, more cheaply. The moment the workers need to share findings and disagree, the mesh is what subagents cannot give you.

## Why it costs more

A subagent spends tokens, then hands you a summary and disappears. That summary is the compression <LessonLink id="I2.1" /> credited delegation for.

A team removes that compression. Every teammate is a separate, full Claude session with its own context window, running for as long as the team does, and messages between them are more tokens on top. Cost scales with the number of active teammates, roughly linearly. Five teammates is close to five sessions' worth of spend, not one.

That is the trade the mesh buys. You pay for genuine parallel reasoning with agents that can argue, and you pay per agent.

## When to care

The tasks that justify the cost share a shape: independent enough to parallelize, but interdependent enough that the workers benefit from talking.

- **Parallel review.** Three teammates on one PR, one per lens (security, performance, tests), synthesized by the lead. A single reviewer drifts toward one kind of issue; three in parallel do not.
- **Competing hypotheses.** A confusing bug handed to several teammates told to disprove each other, like a scientific debate. Sequential investigation anchors on the first plausible theory; adversarial parallel investigation does not.
- **Cross-layer features.** Frontend, backend, and tests, each owned by a teammate who will not touch the others' files.

Reach for a single session or subagents instead when the work is sequential, edits the same files, or carries many dependencies. Coordination overhead is real, and three focused teammates beat five scattered ones.

Two things to know before you go looking for the feature. It is **experimental and off by default**; you turn it on with `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1` in settings or the environment, and it carries live limitations around resuming sessions and shutting teammates down. And the honest default answer to "should I use this today?" is usually no: the work has to genuinely need workers that talk to each other before the token bill and the coordination are worth it.
