# Plan mode vs "make a plan" vs just prompting

## The confusion

Three things produce something that looks like a plan.

You can enter **plan mode** with `Shift+Tab`, as in <LessonLink id="B3.3" />, and Claude researches and proposes without touching your source.

You can stay in Manual mode and simply **ask for a plan**: *"before you change anything, tell me what you would do."* Claude tells you.

Or you can **just prompt**, describe the change, and let Claude do it while you read the diff.

All three end with you looking at a proposal before code lands. The output looks similar enough that people pick one out of habit and never revisit it. Two habits form, both bad. Some learners plan everything, including typo fixes, and mistake the ceremony for rigor. Others never plan, and discover on file six that Claude misunderstood the request on file one.

The difference between the three is not what they produce. It is **what stops Claude from skipping ahead**, and **what it costs you when nothing goes wrong.** That second half is the one people never price.

## Side-by-side

| | Plan mode | "Make a plan" | Just prompting |
|---|---|---|---|
| **How you get it** | `Shift+Tab`, or `/plan` | Ask, in Manual mode | Describe the change |
| **Can Claude edit source?** | **No.** Structurally prevented | Yes, if it decides to | Yes, that is the idea |
| **Guarantee** | Enforced by the mode | Advisory. It is a request | None |
| **Approval step** | Explicit: you approve, then it executes | None. It just continues | None |
| **Context cost** | Highest: exploration and the plan both fill the window | Moderate | Lowest |
| **Time cost when all goes well** | An extra read-and-approve cycle | A paragraph to read | Nothing |
| **What you review** | A plan, then each diff | A plan, then each diff | The diff |
| **Best for** | Unfamiliar code, many files, expensive mistakes | Familiar code, you want the route first | Small, clear, well-understood changes |

Two rows carry the whole lesson.

**"Can Claude edit source?"** is the real distinction, and it is the one the names hide. Asking for a plan is a *request*, and Claude is a model reading context, not a machine obeying configuration. It very often complies. It may also decide the change is simple and get on with it. Plan mode does not rely on compliance: it is enforced, and no amount of misunderstanding on Claude's part will produce an edit.

**"Time cost when all goes well"** is what people never account for. Plan mode always costs an extra cycle, even on the tasks where nothing would have gone wrong. That cost is real and you pay it every time. The question is not "is planning good?" It is "is this task one where the plan will have earned its cycle?"

There is a third input that quietly changes the math. **Cheap undo substitutes for planning.** After <LessonLink id="B3.4" />, you know a bad edit is one `Esc Esc` from gone. If a change is easily reversible, trying it is often cheaper than planning it, and you learn more from the attempt than from the proposal. Planning buys the most where reversal is expensive.

## Which one, as a ladder

Two options do not need a decision tree. Three barely do. Read down until one fits, and stop.

**If you could describe the diff in one sentence → just prompt.**
Typo, log line, rename, one-line guard. Plan it and you will read a plan that tells you what you already knew.

**If it is cheap to undo and you are curious whether it will work → just prompt, and rewind if it does not.**
The experiment is the fastest way to find out. Commit first, then let it try.

**If you know this code, but you want to see the route before Claude drives → ask for a plan.**
No mode switch, no approval cycle. You are not worried about it running off, you just want the sketch.

**If the change spans several files, or you do not know this code, or a wrong edit is annoying to unpick → plan mode.**
This is the case it was built for. Understanding first, and you cannot review a diff in code you do not understand.

**If getting it wrong reaches beyond your machine → plan mode, and read every permission prompt.**
Migrations, deploys, anything pushed. `/rewind` cannot un-send, as <LessonLink id="B3.4" /> is careful to say. Structure is the only protection you have left.

Two inputs run through the whole ladder, and they are not the same thing: **how bad is a mistake**, and **how hard is it to reverse.** A wrong edit in a file you have committed is bad and trivially reversible. A wrong migration is bad and permanent. Only the second one earns full ceremony.

## Scenario walkthroughs

**Rename a variable across three files.**
Just prompt. You could describe the diff in one sentence. Claude does it, you read the diff, done. This is a case where plan mode is a genuine waste of a minute, and where a plan reads as an insult to both of you.

**Add OAuth login to an app you have never touched.**
Plan mode. You do not know the code, the change spans files, and you cannot meaningfully review a diff of an auth flow you have not read. Explore in plan mode, get the plan, fix its misunderstandings while they are still cheap sentences, *then* approve.

**Fix a failing test in a module you wrote last week.**
Just prompt. You know the terrain. Give it the failure output and let it work.

**"I think we should extract this into a service. Not sure it's a good idea."**
Just prompt, then rewind if you hate it. You are not asking whether the change is safe, you are asking whether you like it. A plan describing the extraction will not answer that; seeing the extraction will. Commit first, let Claude try it, look at the result, `/rewind` if it reads worse.

**Change the retry logic in the payments client.**
Ask for a plan, in Manual mode. You know this code well, so plan mode's enforced no-writes buys you little. But the blast radius is money, so you want to see the route before anyone drives. Then read every prompt.

**A database migration.**
Plan mode, and slowly. It is not reversible by any tool in this course. Everything about this lesson is a rehearsal for this case.

**Where plan mode actively slows you down.**
You are adding log lines to trace a bug. You enter plan mode out of good habit. Claude explores the module, writes a plan describing where it will add three log statements, and asks you to approve it. You approve. It adds three log statements.

You spent an extra cycle, filled your context window with exploration you did not need (<LessonLink id="B2.2" />), and learned nothing. Worse, the habit is self-reinforcing: it *felt* careful. Ceremony is not rigor. Rigor is spending the ceremony where the risk actually is, and skipping it where it is not.

**One rung above plan mode.**
When a feature is large enough that even a plan is thin, ask Claude to interview you first, write the result to a spec file, then start a **fresh session** to implement it against that spec. You get a clean context window for implementation and a written artifact to check against. That is beyond this module, but the ladder does keep going.

## Try It

**Environment:** the public repo you cloned in <LessonLink id="B2.1" />. Commit first if you have made changes, so you can reset cleanly.

Pick one small, well-understood change: add a parameter with a default value to an exported function.

1. **Just prompt it.** Ask Claude to make the change. Time it, roughly. Then `git checkout .` to reset.
2. **Ask for a plan.** In Manual mode: *"before you change anything, describe what you would do."* Read it, then tell Claude to proceed. Reset again.
3. **Plan mode.** `Shift+Tab` twice, same request, read the plan, approve with "review each edit manually."

**Success criteria:** all three produced the same change, and you can say which felt proportionate and why. Then answer the real question: **what would have to be different about the task** for your answer to flip? Name the specific property, unfamiliar code, more files, harder to reverse. That property, not the size of the request, is what should drive the choice.

## FAQ

**These three all produce a plan. Which do I actually want?** Match the ceremony to the risk *and* the reversibility. One-sentence diff, just prompt. Familiar code but you want the route, ask for a plan. Unfamiliar code, many files, or expensive to unpick, plan mode.

**Is "make a plan" really different from plan mode?** Yes, in the way that matters. Asking is advisory: Claude may plan, or may decide the task is simple and start editing. Plan mode structurally prevents edits to your source until you approve. One is a request, the other is a guarantee.

**Then why ever just ask?** Because it is free. No mode switch, no approval cycle. When you trust the terrain and only want to see the route, that is exactly enough.

**Does plan mode cost anything?** Yes, and it is worth naming. An extra review-and-approve cycle on every task, plus the context that exploration and the plan consume. On a task that would have gone fine, you paid that for nothing.

**If undo is so cheap, why plan at all?** Because undo only covers Claude's file edits on your machine. Bash side effects, pushes, deploys, and migrations are not reversible, and that is precisely where plan mode earns its cost. Cheap undo substitutes for planning only where undo actually works.

**Should I make plan mode my default?** You can, with `defaultMode` in settings, and some people do. Notice what it will feel like on your next typo fix before you commit to it.
