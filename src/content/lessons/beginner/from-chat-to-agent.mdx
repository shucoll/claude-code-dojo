# From chat to agent

## The problem

You have used AI in a chat window. You paste in some code, describe what you want, read the reply, then copy the suggestion back into your editor and hope it still fits. Every round trip is manual: you are the one moving context in and results out. For a quick question that works fine. For real work, like finding where a bug lives, changing a few files, and running the tests, the copy-and-paste shuffle falls apart fast.

## The concept

Claude Code is **agentic**. Instead of only replying with text, it acts on your project directly. The clearest way to see the difference is to line up the two loops you can be in.

**The chat loop you already know.** You gather the context by hand. You paste it in. You get text back. You apply it by hand. The tool never touches your files, so *you* are the runtime: nothing happens until you copy, paste, and run it yourself.

**The agentic loop.** You describe a goal, and Claude gathers its own context, takes action, and checks the result. It reads files, runs commands, and edits code on your behalf, repeating until the task is done. You are still in charge, but now you steer instead of shuttle.

One analogy: a chat assistant is a consultant who can only mail you advice, while an agent is a capable teammate working in your repo while you watch and redirect.

## How it works

Under the hood, Claude Code wraps a Claude model in an **agentic harness**: the tools, context, and execution environment that let a language model actually *do* things rather than just describe them.

Give it a task and it works through three phases that blend together and repeat:

1. **Gather context:** search the codebase, read the relevant files, check your git state.
2. **Take action:** edit a file, run a command, run the tests.
3. **Verify results:** read the output, decide whether it worked, and choose the next step.

Each result feeds the next decision, so Claude chains many steps together and course-corrects as it learns. The loop adapts to the request. A question about your code might only need step one; a bug fix cycles through all three several times.

Those actions come from **tools**. Informally they fall into a few buckets: reading and editing files, searching, running shell commands, and looking things up on the web. Without tools a model can only produce text. Tools are exactly what turn "chat" into "agent." (You will meet the full tool set later in the course; here it is enough to know why they matter.)

To make this concrete, picture a failing test. In a chat tool you would copy the error, paste in the files you think are relevant, and relay fixes back and forth. With Claude Code you say:

```bash
claude "the checkout test is failing, find out why and fix it"
```

From that one sentence, Claude might run the test suite to see what fails, read the error, search for the source files, read them to understand the code, make the fix, then run the tests again to confirm. You never told it which files to open or which commands to run. That is the agentic loop deciding each step from what the previous one revealed.

And you are part of the loop. Claude works on its own but stays responsive: press `Esc` to stop it, or type a correction to nudge it mid-task without starting over.

## FAQ

**How is this different from pasting code into a chat?** In a chat you do the legwork: gathering files, applying answers, running things yourself. Claude Code closes that loop for you. It reads your files, makes the change, and checks the result, so your job shifts from shuttling text to steering a teammate.

**What does "agentic" actually mean?** It means the model can take actions through tools (read, edit, run) and react to what those actions return, not just emit text. That feedback-driven cycle, decide then act then observe then repeat, is the "agent" part.

**Does it act without asking me?** Not by default. Claude asks before it edits files or runs shell commands, and it always asks before actions that reach outside your project, like deleting data or calling a live service. You approve each step until you deliberately choose to loosen that. You are steering the whole time.

**Can it break my computer?** It is scoped to the directory you start it in, so it works inside your project, not across your whole machine. On top of that, every file edit is snapshotted first, so you can rewind changes. The actions a snapshot cannot undo, the ones that touch systems outside your files, are exactly the ones it stops to ask about first.
